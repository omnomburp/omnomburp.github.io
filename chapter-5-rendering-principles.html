Chapter 5

Rendering is the process of turning scene data into a trustworthy image

By this point you have seen transforms, spaces, normals, and shader stages separately. Rendering is where those ideas cooperate. The GPU has to decide which fragments are visible, how they sample data, and what final color they should contribute to the screen.

Rasterization

Triangles become fragments that can be tested and shaded.

Depth

Visibility is usually decided by keeping the fragment with the smallest depth value.

Sampling and materials

Textures, shading models, and color transforms shape the final appearance.

Mental model

A frame is a sequence of decisions about geometry, visibility, and color

1

Geometry is projected

After transforms and projection, triangles are ready to be turned into screen-space coverage.

2

Fragments compete for visibility

Depth testing usually decides which fragment represents the front-most visible surface.

3

4

Color needs context

Materials, lighting, tone mapping, and display space all influence the final image that reaches the eye.

Reinforcement

Repeat the frame story as several tiny render decisions

Rendering feels large because many small decisions happen quickly. Breaking the frame into compact examples helps you recognize the same logic when the full pipeline starts to feel busy.

Coverage depends on sample positions

The rasterizer is not guessing which pixels are “kind of inside.” It tests actual sample locations against triangle coverage.

covered(sampleCenter)

Interpolated values come from triangle weights

One fragment inherits a blend of the vertex data, which is why colors, UVs, and normals can vary smoothly inside a face.

v = a*A + b*B + c*C

Depth keeps the front-most winner

Two surfaces can want the same pixel. The depth test decides which fragment actually gets to update the frame buffer.

if (z < zBuffer) write

Sampling turns discrete texture data into a visible color

The source texture is a grid, but the renderer still has to answer “what color should this one fragment receive?”

color = texture(tex, uv)

Triangle story

Before a fragment exists, the GPU still has to clip, cull, cover pixels, and interpolate data

The phrase “rasterize the triangle” hides several important steps. The GPU does not jump directly from three transformed vertices to a final pixel color. It first has to decide which triangle survives, which part stays inside the view, which pixel centers count as covered, and how attributes should blend inside that coverage.

Visual explainer

The triangle is clipped to the view, back faces can disappear, pixel centers determine coverage, and barycentric weights blend the vertex data inside the surviving area.

  1. Clip triangles against the view volume

    If part of a triangle falls outside the clip region, the GPU trims it before coverage is evaluated.

  2. Optionally cull back-facing triangles

    Faces pointing away from the camera are often thrown out early so they never become fragment candidates.

  3. Test pixel centers for coverage

    A triangle does not own every pixel it touches vaguely. The rasterizer uses exact sample positions to decide coverage.

  4. Interpolate attributes with barycentric weights

    The covered sample inherits a blend of the triangle’s vertex data, which is why one varying can change smoothly across the surface.

Why clipping exists

It keeps the pipeline from trying to rasterize geometry that should never appear inside the current view.

Why barycentric weights matter

They are the hidden math behind smooth color, UV, and normal interpolation across one triangle.

Visibility story

One pixel can receive multiple fragments, so the depth buffer keeps score

Rendering errors often feel mysterious because the final image hides the competition that happened per pixel. This view slows that down and shows why depth testing decides which surface gets to speak for a screen sample.

Visual explainer

Left: overlapping screen coverage. Middle: fragment candidates for one pixel. Right: the depth and color buffers keep the nearest winner.

  1. Rasterization can produce several fragment candidates for one pixel

    If two triangles overlap on screen, both can generate a fragment for the same pixel center.

  2. Each candidate carries both color and depth

    The color says what the surface would contribute. The depth says how far that surface is from the camera.

  3. The depth test compares the incoming fragment to the stored winner

    In the common case, the fragment with the smaller depth wins and replaces the older value in both buffers.

  4. The color buffer shows only the winner unless blending changes the rule

    Opaque rendering is mostly a one-winner story. Transparency and blending need extra rules because more than one layer can matter.

Depth buffer memory

Think of it as one stored depth number per pixel. Every new fragment has to beat that number before it can update the image.

Useful debugging question

If the wrong surface appears, ask whether the bug is in fragment generation, depth values, or the test itself before blaming the shader.

Step by step

What the GPU does during a single draw

  1. Transform vertices into clip space

    The earlier chapters handle this part: model, view, and projection establish where the triangles go.

  2. Rasterize triangles into fragments

    The GPU determines which pixel-sized regions are touched by each triangle.

  3. Run fragment shading

    Visible candidates receive material, lighting, and texture logic to produce a color.

  4. Resolve visibility and write the result

    Depth and blend rules decide whether the fragment updates the frame buffer.

Why depth matters

Without depth testing, the final picture often depends on draw order rather than real spatial visibility.

Why sampling matters

Textures and screens are grids, so aliasing and filtering are unavoidable topics in real renderers.

Why color management matters

If your math happens in the wrong color space, the image can look muddy or too dark even when the geometry is correct.

Transparency is a separate visibility problem

Alpha-blended surfaces often need sorted draw order because the depth buffer alone cannot combine partially transparent layers correctly.

Framebuffers

A framebuffer is a render target that does not have to be the screen

So far, every draw call writes to the default framebuffer — the image that appears on your monitor. But the GPU can draw into a texture-backed framebuffer object (FBO) instead. That texture can then be sampled in a later pass, which is the foundation of shadow maps, post-processing, reflections, and deferred shading.

Pass 1 renders the scene into an off-screen texture. Pass 2 reads that texture and draws a full-screen quad to the screen. The texture bridges the two passes.

The default framebuffer is the screen

Every draw call you have seen so far writes directly to the display. The GPU picks the default framebuffer when no FBO is bound.

An FBO redirects rendering to a texture

Create a framebuffer object, attach a texture as the color target, and bind it before drawing. The scene renders into the texture instead of the screen.

The texture becomes a readable result

Once the off-screen pass is done, you unbind the FBO and bind the texture as a regular sampler input. Later shaders can read the result as if it were any other texture.

Depth attachments work the same way

Shadow maps store depth values in an FBO's depth attachment. The shadow pass writes depth from the light's viewpoint; the scene pass reads it to decide which fragments are in shadow.

// pass 1: render scene into FBO
bindFramebuffer(fbo);
drawScene();

// pass 2: use the result as a texture
bindFramebuffer(null);   // back to screen
bindTexture(fbo.colorAttachment);
drawFullscreenQuad();

Resolution is independent

The off-screen texture can be any size. Shadow maps are often 1024×1024 or 2048×2048 regardless of screen resolution. Post-processing can render at half resolution for speed.

Multiple attachments

A single FBO can have multiple color attachments. Deferred shading uses this to write position, normal, and albedo into separate textures in one pass.

Multi-pass rendering

Complex effects are built by chaining render passes

A single frame in a modern engine often involves several render passes, each targeting a different framebuffer. One pass captures shadow depth, another draws the lit scene, another applies bloom or blur, and a final composite pass assembles the result for the screen. Each pass reads textures produced by earlier passes.

A typical frame: shadow pass writes depth from the light, scene pass reads it for shadows, post-processing blurs bright areas, and the final pass composites everything to the screen.

Shadow pass

Render the scene from the light's viewpoint into a depth-only FBO. The result is a shadow map — a texture of depth values that the scene pass uses to test visibility from the light.

Scene pass

Render the scene from the camera's viewpoint into an off-screen color FBO. The shader samples the shadow map to darken occluded fragments. The result is an HDR color texture.

Post-processing pass

A full-screen quad shader reads the scene texture, extracts bright areas, and blurs them to create bloom. Other effects — tone mapping, color grading, anti-aliasing — happen here too.

Final composite

The last pass writes to the default framebuffer (the screen). It combines the post-processed result with any UI overlays and applies the display transfer function.

// frame structure
pass 1: shadow depth  → FBO_shadow
pass 2: lit scene     → FBO_scene
        (reads FBO_shadow)
pass 3: bloom blur    → FBO_bloom
        (reads FBO_scene)
pass 4: composite     → screen
        (reads FBO_scene + FBO_bloom)

Deferred shading

Instead of lighting every fragment during the scene pass, deferred shading writes geometry data (position, normal, albedo) into a G-buffer with multiple FBO attachments. A later full-screen pass lights each pixel using the G-buffer. This decouples geometry from lighting and handles many lights efficiently.

Cost of passes

Each pass has overhead: switching FBOs, clearing buffers, and issuing draw calls. Engines minimize passes by combining work when possible, but the flexibility of multi-pass rendering usually outweighs the cost.

Live lab

Turn depth testing on and off to see visibility break

Two cubes overlap in screen space. With depth testing enabled, the front-most fragments win. With depth testing disabled, draw order wins instead, which is not the same thing. This is one of the fastest ways to build intuition for why the depth buffer exists.

  • Turn depth testing off first. The later draw call starts to overwrite fragments that should stay hidden.
  • Reduce layer separation to make the overlap more obvious.
  • Turn depth testing back on and watch the front cube reclaim the correct visible regions.

When depth testing is disabled, the image reflects draw order instead of true visibility.
