Chapter 1

Vectors, bases, and matrix transforms

Computer graphics begins with a simple question: how do we describe points and directions in space? Vectors answer that question. Bases tell us what the coordinate axes mean. Matrices tell us how that local frame can rotate, stretch, and, later, with one extra coordinate, translate.

What you should leave with

Confidence reading a point as weighted basis directions rather than a mysterious pair of numbers.

Main idea

A matrix changes how a shape is interpreted inside a coordinate frame.

Why it matters

Everything later in graphics depends on moving data between spaces without losing track of meaning.

Start here p = x i + y j

Read coordinates as weights on basis directions, not as unexplained numbers.

Then change the frame p' = M p

A matrix changes how the same local point is rebuilt in space.

Next question point or direction?

We will learn translation after we separate those two cases clearly.

Foundations

Separate locations, offsets, and directions before you touch matrices

Graphics reuses the same tuple syntax for several different geometric meanings. That is why the math can feel slippery at first. The numbers matter, but the thing those numbers describe matters just as much.

A point owns a location, an offset spans between points, and a direction keeps orientation without owning one fixed position.

Point

A point answers “where is it?” In a game, playerPos is a point, so translation should move it.

Offset vector

An offset answers “how do I get from one point to another?” In a game, enemyPos - playerPos is that kind of arrow.

Direction

A direction answers “which way?” In a game, the stick direction or aim direction should not gain a new location just because the player moved.

  1. A point and a direction can share the same raw numbers

    (1, 0, 0) could mean “the point one unit along x from the origin” or “the direction pointing along x.” The tuple alone does not settle that meaning.

  2. Subtracting two points gives an offset

    If B - A points from one location to another, the result is a reusable offset. It is not tied to one absolute origin the way a point is.

  3. This distinction is why homogeneous coordinates appear later

    When we add w, we are not inventing a random extra number. We are giving the pipeline a way to tell “location” apart from “pure direction.”

Good habit

When you read a graphics value, ask “is this a point, an offset, or a direction?” before you ask what its numbers are.

Why chapter 1 slows down here

If these meanings blur together now, later topics like normals, camera transforms, and lighting will feel arbitrary even when the code is correct.

Game Intuition

Vectors already run a top-down game even if the engine hides the math

Move a character, aim at an enemy, launch a projectile, or apply knockback and you are already using vectors. Games are one of the clearest places to see what each vector means in motion.

The player marker is a point. The orange arrow is velocity. The cyan arrow is aim direction. The dashed arrow to the enemy is the offset vector built from subtraction.

Player point

The character’s current map location is a point.

playerPos

Movement vector

The stick picks a direction, then speed gives that direction a length.

velocity = moveDir * speed

Target offset

Subtract two points to get the arrow from one actor to another.

toEnemy = enemyPos - playerPos

Frame update

Move the point by adding this frame’s displacement.

playerPos += velocity * dt

  1. The player marker is a point in the world

    The circle has one location. That makes it a point, not just a free-floating arrow.

  2. Velocity says how the point will change next

    The orange arrow has both direction and length, so it can mean “move this way at this speed.”

  3. Aim is a direction, not a location

    The cyan arrow tells the projectile which way to travel. It should rotate with the player but not inherit a second world position of its own.

  4. Subtract positions to get chase, aim, and knockback vectors

    The dashed arrow from player to enemy is the kind of vector a game uses for steering, targeting, and many AI behaviors.

Why normalization shows up in games

If you only want “which way?”, normalize the vector first. That removes length so the value keeps direction but not speed.

Why this helps with graphics too

The same meanings keep showing up later as light directions, view directions, camera offsets, and local basis axes.

Vector Relations

Dot compares directions. Cross builds what is perpendicular.

These two operations keep showing up because graphics constantly asks two questions: “how aligned are these directions?” and “what direction sticks out of this surface?” The dot product answers the first. The cross product answers the second.

Left: dot tests whether a target is in front of the player. Right: crossing two triangle edges produces a surface normal.

Dot product

With unit vectors, dot is an alignment score: 1 means same way, 0 means sideways, and negative means opposite.

dot(a, b)

Graphics uses for dot

It powers diffuse lighting, front-versus-behind tests, and projections onto an axis.

brightness = max(dot(n, l), 0)

Cross product

Cross returns a vector perpendicular to both inputs, so it can turn two edge directions into one surface-facing direction.

cross(a, b)

Graphics uses for cross

It builds triangle normals, camera side axes, and winding-based facing tests.

n = normalize(cross(e1, e2))

  1. Normalize before reading dot as pure alignment

    If lengths are still mixed in, the value is no longer just about angle. In lighting, that usually creates the wrong intuition.

  2. Dot answers a yes-no style gameplay question very well

    Is the enemy in front of the player? Is the flashlight hitting this patch head-on? Dot is the fast alignment test behind both questions.

  3. Cross depends on order

    cross(a, b) points the opposite way from cross(b, a). That is why triangle winding changes which side of a surface is considered the front.

Unit vectors dot(a, b) = cos(theta)
Order matters cross(b, a) = -cross(a, b)

Why this belongs in graphics

Chapter 3 will use dot for light response constantly. Cross is how surfaces and camera frames recover a perpendicular direction from geometry.

Reinforcement

See the same vector tools do four different jobs

A concept usually clicks when it survives a few context changes. These short examples reuse the same vector ideas in movement, steering, facing tests, and surface construction so the chapter does not leave them behind after one explanation.

Subtract points to get a useful offset

The same subtraction that gives a chase arrow can also drive pickups, AI steering, and knockback direction.

toPickup = pickup - player

Normalize when you only want direction

The raw input arrow can get longer or shorter, but the unit vector keeps only orientation, which is what movement and lighting often want.

moveDir = normalize(input)

Dot answers “is it in front?”

One direction says where the player is facing. Another says where the target is. Dot compares them without needing a full angle calculation.

dot(forward, toTarget)

Cross turns two edges into a normal

Triangles do not store “front” as a magical property. Two edge directions can build the surface-facing normal explicitly.

normal = cross(e1, e2)

Mental model

Think of coordinates as instructions for rebuilding a point from basis directions

The numbers inside a vector are not meaningful by themselves. They only become meaningful after you decide what the basis directions are. In graphics, changing the basis is often just as important as changing the point.

1

The raw numbers are not the picture yet

A coordinate like (2, 1) only becomes meaningful after you decide what the basis directions are.

2

A basis picks the local language

The usual 2D basis is “one step right” and “one step up,” but you are free to rotate or stretch that language.

3

Coordinates are basis weights

If a point is (2, 1), read that as “two copies of the x basis plus one copy of the y basis.”

4

A matrix changes that reconstruction rule

Rotation and scale work by changing the basis itself. Translation is special enough that we will slow down and learn it next.

Read A Matrix

In graphics, the columns of a matrix are often the new basis directions

This is the jump many beginners need. A matrix is not just a box that “does multiplication.” It packs the basis vectors that tell you how local coordinates are rebuilt in another frame.

The matrix columns are the new basis directions. The unit square follows those columns into a transformed parallelogram.

First column

It tells you where the local x-axis lands after the transform. Feed in (1, 0) and you recover that column.

Second column

It tells you where the local y-axis lands. Feed in (0, 1) and you recover the second column.

Why this explains scale, rotation, and shear

If those column vectors change length, angle, or perpendicularity, the rebuilt shape stretches, rotates, or shears with them.

M = [ i'_x  j'_x ]
    [ i'_y  j'_y ]

p_local = [ x ]
          [ y ]

p_world = M p_local
        = x i' + y j'

Beginner-friendly reading

Matrix multiplication here means “take x copies of the first column and y copies of the second.” It is the same basis-weight story, just packed into one object.

Rebuild A Point

Keep the weights fixed and let the basis move first

Before translation or long matrix chains, keep one sentence in your head: the point is rebuilt from basis directions. The animation below keeps the weights fixed while the basis slowly rotates and stretches.

Visual explainer

Same weights every frame. Different basis every frame. The point is rebuilt from the moving basis.

  1. Start with one local coordinate

    Imagine the point (1.2, 0.65). Read that as “take 1.2 copies of i and 0.65 copies of j.”

  2. Build the point from those two contributions

    The orange segment shows the contribution from i. The green segment adds the j contribution to reach the final point.

  3. Move the basis, not the weights

    As the basis rotates and stretches, the same weights rebuild a different final position. That is the core visual idea behind a matrix transform.

Plain-language sentence

“The point moved because the frame I used to rebuild it changed.”

What to notice

The weights stay constant throughout the animation. Only the basis directions and their lengths are changing.

Affine Bridge

Linear transforms keep the origin fixed, so translation must be added as an offset

Rotation, scale, and shear are linear: they rebuild coordinates around the current origin. Translation is different because it moves that origin itself.

A linear transform keeps the origin fixed. An affine transform adds translation and shifts the whole frame with it.

Linear part

A p can rotate, scale, or shear a value, but it always sends the zero vector to zero.

Why that matters

If the origin stays fixed, a pure matrix multiply in ordinary coordinates cannot describe “move everything two units right.”

Affine form

An everyday object transform is usually p' = A p + t: a linear basis change followed by one shared offset.

Homogeneous trick

Homogeneous coordinates package that offset into the same matrix chain so the GPU can compose everything uniformly.

linear:  p' = A p
affine:  p' = A p + t

A * 0 = 0

That last line is the reason
translation needs special treatment.

What changes conceptually

First rebuild the local value from basis directions. Then, if the value is a point, add the frame’s translation offset.

What stays the same

The linear part can affect both points and directions. The offset should only affect points, which is exactly what w lets the pipeline express.

Translation And W

Translation needs one extra coordinate because points and directions should not behave the same way

Rotation and scale can be understood as basis changes. Translation is different: it should move points, but it should not move pure directions. Graphics handles that by adding one extra coordinate called w, which lets one matrix chain express both the linear part and the optional translation offset.

Visual explainer

Same local value, same translation vector. The point moves because w = 1; the direction stays put because w = 0.

  1. A point should move with translation

    If a point marks a location on an object, translating the object should move that location too.

  2. A direction should keep its meaning

    If a value means “this way,” shifting the whole object should not change that direction or its length.

  3. Homogeneous coordinates add one more slot

    Write points as (x, y, z, 1) and directions as (x, y, z, 0). Now the matrix chain can tell those cases apart.

  4. The translation offset is multiplied by w

    When w = 1, the offset is applied. When w = 0, the translation contribution disappears.

Point p_h = (x, y, z, 1)
Direction d_h = (x, y, z, 0)

Order Matters

Applying the same transforms in a different order produces a different motion

This idea is easier to trust once you see it. Both panels below use the same changing scale and rotation values, but the highlighted corner traces a different path because the second transform always acts on the result of the first. Read the chain right to left: whichever transform sits closest to the point acts first.

Visual explainer

Left: scale then rotate. Right: rotate then scale. The highlighted corner follows a different orbit in each case.

Scale then rotate

The shape stretches along its local axes first, then the already-stretched result is turned as a whole.

Rotate then scale

The shape turns first, so the later scale happens along axes that have already been rotated.
