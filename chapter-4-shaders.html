Chapter 4

Shaders let the GPU reshape geometry and color the final picture

A shader is a small program that runs many times on the GPU. The important beginner move is not memorizing syntax. It is learning which stage is responsible for which kind of work.

Vertex shader

Good for moving geometry, preparing varyings, and expressing transforms.

Fragment shader

Good for per-fragment color, lighting, texturing, and pattern generation.

Shared data

Attributes feed vertices, uniforms stay constant for a draw call, and varyings get interpolated.

Vertex output gl_Position = P V M p
Shared parameter uniform = same for all vertices
Interpolated value varying -> fragment

Mental model

Think in jobs: move geometry first, shade fragment candidates later

1

Attributes belong to vertices

Position, normal, color, and texture coordinates usually enter the pipeline attached to each vertex.

2

The vertex shader prepares geometry

It decides where each vertex goes and which values should be passed forward for interpolation.

3

Rasterization fills the gaps

The GPU turns transformed triangles into many fragments and blends the varying values across them.

4

The fragment shader finishes the look

It receives interpolated values and computes a color for each fragment candidate before the remaining per-sample tests and blending finish the job.

Step by step

Separate vertex work from fragment work clearly

vertex shader
  read position
  bend or move the mesh
  write gl_Position
  pass varyings forward

fragment shader
  read interpolated varyings
  compute lighting or pattern
  write final color

Useful rule of thumb

If the effect depends on the shape of the mesh, start by asking whether it belongs in the vertex shader.

  1. Move vertices in the vertex shader

    Changing position here means the mesh itself changes before triangles are assembled.

  2. Pass values along as varyings

    UVs, normals, colors, and custom values can be handed to the fragment stage for interpolation.

  3. Use uniforms for shared controls

    Time, camera matrices, light direction, and slider values are usually uniforms.

  4. Shade per fragment

    Patterns and lighting belong here because they need high spatial detail across the surface.

Data path

A draw call turns raw arrays into shader inputs before the GPU can do any shading at all

Shader stages are only half the story. Before the vertex shader can run, the CPU has to upload vertex data, describe how to read it, and issue a draw call. This is the missing bridge between “shader code exists” and “the GPU can actually start rendering.”

Step through the draw call

CPU arrays begin on the CPU. They only become shader inputs after upload, attribute wiring, and a draw call.

Current step

Start with raw vertex data on the CPU

Positions, UVs, normals, and indices usually begin as plain arrays of numbers in application memory.

What changes next

Those numbers have to be uploaded into GPU buffers before the draw call can reference them.

Key term

CPU memory is the host-side copy. The GPU still knows nothing about these arrays until you upload them.

VBO intuition

Think of a vertex buffer as raw GPU-side storage for per-vertex data, not as a shader by itself.

Index buffer intuition

An index buffer is the reuse plan. It tells the draw call which existing vertices should be assembled into triangles.

VAO intuition

In modern GL-style APIs, a vertex array object is the saved recipe for how those buffers map into attributes.

GPU simulation

Fluid-like particles show the kind of job compute shaders are built for

Vertex and fragment shaders are tied to drawing geometry. Compute shaders are different: they update arbitrary state first, then a later render pass visualizes that state. This chapter still runs on a WebGL baseline, so the live scene below teaches the same parallel idea with a lightweight particle fluid and compute-style pseudocode rather than a true compute dispatch.

One invocation per particle

Particle motion is repetitive work. Every element follows the same update rule, which is exactly what makes it parallel-friendly.

Update pass before draw pass

Compute-style work updates positions and velocities first. A later render pass reads that state and draws the visible spheres.

Why it often beats the CPU

The GPU can update many particles at once and keep the state on the device instead of bouncing it back through CPU-side loops every frame.

Click or drag inside the tank to inject motion. Then raise the particle count and keep an eye on the serial CPU work versus the compute-style dispatch summary. The visual stays simple on purpose so the lesson is about the job split: update state first, render it second.
