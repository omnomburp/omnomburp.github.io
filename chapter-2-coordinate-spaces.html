Chapter 2

Coordinate spaces are the grammar of a scene

A graphics pipeline does not move a model directly from “data” to “pixels.” It moves that model through several spaces, each of which makes one job easier. Understanding those spaces is how you stop getting lost in transformation code.

Object space

The model’s own local frame, where the shape is easiest to describe.

World and view space

Place the model in a scene, then reinterpret the scene from the camera’s point of view.

Clip space

The projection output where visibility is tested against w. It is not the same thing as the final screen image.

Object to world p_world = M p_object
World to view p_view = V p_world
Clip to NDC p_ndc = p_clip / w

Space Contract

A position without its space is only half a variable name

In graphics, coordinates behave like typed data. The tuple is the value, and the current space is the type. Dropping the space name is like dropping units from a physics equation.

The same geometric vertex is rewritten under different space contracts, so the numbers can change meaning even before the shape changes.

Object position

(1, 0, 0) means “one local unit along the model’s x-axis.”

World position

The same numbers now mean “one world unit from the global origin,” which can be a completely different place.

View position

Now the camera is the reference frame. Left, right, near, and far are camera-relative.

Clip position

After projection the values are no longer ordinary scene distances. They are homogeneous pre-divide coordinates used for clipping.

  1. Same vertex, new description

    The GPU is not inventing four different vertices. It is rewriting the same geometric point so each stage can do its own job.

  2. Each transform has an input contract

    The model matrix expects object-space input. The view matrix expects world-space input. The projection matrix expects view-space input.

  3. Most graphics bugs are really space-mismatch bugs

    If lighting, culling, or projection code receives data from the wrong space, the numbers can still look plausible while the picture is wrong.

Treat spaces like units

A world-space normal and a view-space normal are as meaningfully different as meters and inches. Both may be valid, but you cannot silently swap them.

Why chapter 2 matters so much

Shading, cameras, culling, and rasterization all assume you know which space a value currently lives in. That assumption starts here.

Game Intuition

A chase camera is a coordinate-space machine running every frame

In a game, the same ship or player can be described in several useful ways at once: local model coordinates for the mesh, world coordinates for the level, and camera-relative coordinates for rendering.

Left: the ship in local model space. Middle: the same ship placed into the arena. Right: the same arena rewritten relative to the chase camera.

Model-local mesh

The ship is easiest to define around its own pivot before the level matters.

shipLocal

World placement

The same ship gets one position and facing inside the shared arena.

shipWorld = M * shipLocal

Camera-relative view

Subtract the camera pose and rewrite the scene around the chase camera.

pickupView = V * pickupWorld

Next step

Only after the camera-relative rewrite does projection prepare the final screen mapping.

screen = project(view)

  1. Object space is where the asset is authored

    The ship mesh sits around its own origin so rotating, animating, and attaching effects remain simple.

  2. World space is where gameplay happens

    The chase camera, pickups, and enemies all need shared positions before the game can reason about distance and collision.

  3. View space rewrites the arena around the camera

    The camera is easier to render from once the whole world is expressed relative to its frame.

  4. The same object survives every rewrite

    The ship is not duplicated three times. Only its description changes from local, to world, to camera-relative coordinates.

Useful game habit

If an object renders in the wrong place, ask whether the bug happened before or after the camera rewrite instead of treating the full pipeline as one step.

Why this matters beyond games

The same coordinate bookkeeping shows up in CAD viewers, simulations, map tools, and any other 3D renderer with a camera.

Mental model

Each space exists because it makes one transformation problem easier

Instead of asking “why are there so many spaces?”, ask “what job becomes easier in this space?” The pipeline is a sequence of coordinate systems chosen for convenience, not a sequence of arbitrary math tricks.

1

Object space keeps modeling simple

A chair can be described around its own origin, regardless of where it later sits in a room.

2

World space builds the scene

Models gain meaningful placement relative to one another: left, right, above, behind, and so on.

3

View space centers the camera

The camera transform is easier to reason about once the world is expressed relative to the eye.

4

Clip space prepares the divide

Projection first produces a homogeneous clip coordinate. Only after dividing by w do you land in normalized device coordinates.

Reinforcement

Reuse the spaces in concrete rendering and gameplay tasks

Coordinate spaces stop feeling arbitrary once you see why different jobs prefer different frames. These compact examples repeat the chapter’s spaces in the kinds of tasks that actually show up in an engine.

Object space keeps attachments simple

A muzzle socket or hand socket is easiest to author in the model’s own local frame, then transform with the rest of the asset.

socketWorld = M * socketLocal

World space is where distance makes sense

AI range checks, collisions, and navigation want objects expressed in one shared scene frame before any camera rewrite happens.

enemyWorld - playerWorld

View space rewrites around the camera

The target is still the same object, but the useful question becomes “where is it relative to the camera now?”

targetView = V * targetWorld

NDC and pixels are screen-facing spaces

After the divide by w, the point finally lives in the fixed screen-ready box that the viewport maps into pixels.

pixel = viewport(ndc)

Where Transforms Act

Each matrix acts between two spaces, not everywhere at once

A lot of graphics confusion comes from treating positions like one universal thing. They are not. The position keeps changing meaning as each matrix rewrites it into a new space.

Visual explainer

The same vertex is redrawn in each space, and the labels show which transform performs each rewrite.

  1. The model matrix acts on object-space positions

    It takes a vertex from the model’s own local frame and places it into the shared scene.

  2. The view matrix acts on world-space positions

    It rewrites the whole scene relative to the camera, so the eye behaves like the origin.

  3. The projection matrix acts on view-space positions

    Its output is clip space: a homogeneous coordinate that still carries w.

  4. The divide by w is its own step

    No new matrix is applied there. The GPU divides the clip coordinate by w to reach normalized device coordinates.

Good naming habit

If a variable is named viewPosition, that name should tell you which transforms have already happened and which have not.

Good debugging habit

When something disappears, ask which step likely pushed it out of the valid region instead of treating the full chain as one black box.

View Matrix

The camera is easier to reason about once the world is rewritten around it

Beginners often imagine the view matrix as “moving the camera forward.” In math terms it does the opposite: it rewrites world-space points so the camera behaves like the origin looking down its forward axis.

In world space the camera has a pose. In view space the eye is fixed and the world moves inversely around it.

  1. Start from a camera pose in world space

    The camera has a position and orientation just like any other object in the scene.

  2. Invert that pose to build the view matrix

    If the camera moved right in world space, the view transform makes the whole world appear to move left instead.

  3. After the rewrite, the eye is at the origin

    That makes projection easier because the visible volume can now be defined relative to one fixed camera frame.

Camera pose cameraPose
View matrix V = inverse(cameraPose)

Practical reading

Do not memorize “the camera moves.” Read it as “world data is being re-expressed in camera-relative coordinates.”

Clip Space

Clip space is a temporary homogeneous checkpoint, not the screen

This is the term that most often gets blurred together with NDC or “screen space.” Clip space is earlier than both. It is the projection output before the perspective divide.

View space

The point is camera-relative, but projection has not happened yet.

p_view = (x_v, y_v, z_v)

Clip space

Projection has happened. The point is now homogeneous and still carries w.

p_clip = (x_c, y_c, z_c, w_c)

NDC

The same point after dividing the clip coordinate by w.

p_ndc = p_clip / w_c

Viewport

No new 3D transform here. The GPU maps NDC into pixel coordinates on the screen.

pixel = viewport(p_ndc)

Visual explainer

Projection creates clip space. Dividing by w creates NDC. The viewport then maps that normalized box into pixels.

View point

(0.00, 0.00, 0.00)

Clip point

(0.00, 0.00, 0.00, 1.00)

NDC point

(0.00, 0.00, 0.00)

Viewport pixel

(0, 0)

  1. Projection outputs a homogeneous clip coordinate

    You still have four numbers there, and w is one of the important ones.

  2. Visibility is tested against w

    For a point to survive clipping, the clip-space values have to stay inside the range -w to +w.

  3. NDC is the post-divide coordinate

    After dividing by w, the point lands in the fixed [-1, 1] box. That new post-divide value is NDC, not clip space.

  4. The viewport maps NDC to pixels

    Only after the divide does the GPU map the normalized box into actual screen coordinates. The viewport is a screen mapping step, not another 3D transform.

Clip test -w <= x_c, y_c, z_c <= w
NDC step p_ndc = p_clip / w
Viewport step pixel = viewport(p_ndc)

Worked Example

Track one concrete vertex and notice what changes at each rewrite

A beginner-friendly way to read the pipeline is to keep one vertex in mind and narrate what each stage changes: reference frame, not identity.

One highlighted vertex keeps its identity while its coordinates are rewritten through object, world, view, clip, and NDC stages.

  1. Object space: a corner of the model

    p_object = (1, 0, 0, 1) means “one local unit along the model’s x-axis.”

  2. World space: same corner, now placed in the scene

    After a model transform, that corner might become p_world = (4, 1, -5, 1). The vertex did not change identity; only the frame used to describe it changed.

  3. View space: same corner relative to the camera

    If the camera is only translated to (2, 0, 3) in world space, the same point can become p_view = (2, 1, -8, 1) after the view rewrite.

  4. Clip and NDC: same corner prepared for visibility and screen mapping

    Projection turns the view-space point into p_clip. Dividing by w turns that into p_ndc, which the viewport can finally map to pixels.

p_object -> M -> p_world
p_world  -> V -> p_view
p_view   -> P -> p_clip
p_clip   -> /w -> p_ndc

What stays constant

It is still the same vertex on the same triangle. The numbers change because the coordinate system changes.

Why beginners get lost

If you collapse all four readouts into the single word “position,” you erase the main information needed to debug the pipeline.

Step by step

Follow a single vertex through the pipeline

  1. Begin in object space

    The vertex lives in the model’s own local coordinates. Nothing about the scene or camera matters yet.

  2. Apply the model matrix

    This moves the model into world space: translation, rotation, and scale usually happen here.

  3. Apply the view matrix

    The whole world is re-expressed from the camera’s frame so “forward,” “left,” and “up” become camera-relative.

  4. Apply projection

    Perspective or orthographic projection shapes the visible volume into clip space. That clip coordinate still carries w. Only after dividing by w do you get normalized device coordinates.

The full chain

Most tutorials write this as P * V * M * p_h. Read it right to left as a sequence of spaces ending in a divide by w.

Clip versus NDC

Clip space is the projection output that still carries w. NDC is the new coordinate you get after dividing by w. The viewport then turns NDC into pixels.

Why names matter

If a variable is called worldPosition, that name should tell you what operations are still valid on it.

Debugging habit

When geometry disappears, ask which stage likely pushed it out of the visible volume.
