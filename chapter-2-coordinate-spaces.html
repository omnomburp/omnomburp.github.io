Chapter 2

Coordinate spaces are the grammar of a scene

A graphics pipeline does not move a model directly from “data” to “pixels.” It moves that model through several spaces, each of which makes one job easier. Understanding those spaces is how you stop getting lost in transformation code.

Object space

The model’s own local frame, where the shape is easiest to describe.

World and view space

Place the model in a scene, then reinterpret the scene from the camera’s point of view.

Clip space

The projection output where visibility is tested against w. It is not the same thing as the final screen image.

Object to world p_world = M p_object
World to view p_view = V p_world
Clip to NDC p_ndc = p_clip / w

Mental model

Each space exists because it makes one transformation problem easier

1

Object space keeps modeling simple

A chair can be described around its own origin, regardless of where it later sits in a room.

2

World space builds the scene

Models gain meaningful placement relative to one another: left, right, above, behind, and so on.

3

View space centers the camera

The camera transform is easier to reason about once the world is expressed relative to the eye.

4

Clip space prepares the divide

Projection first produces a homogeneous clip coordinate. Only after dividing by w do you land in normalized device coordinates.

Where Transforms Act

Each matrix acts between two spaces, not everywhere at once

A lot of graphics confusion comes from treating positions like one universal thing. They are not. The position keeps changing meaning as each matrix rewrites it into a new space.

Visual explainer

The same vertex is redrawn in each space, and the labels show which transform performs each rewrite.

  1. The model matrix acts on object-space positions

    It takes a vertex from the model’s own local frame and places it into the shared scene.

  2. The view matrix acts on world-space positions

    It rewrites the whole scene relative to the camera, so the eye behaves like the origin.

  3. The projection matrix acts on view-space positions

    Its output is clip space: a homogeneous coordinate that still carries w.

  4. The divide by w is its own step

    No new matrix is applied there. The GPU divides the clip coordinate by w to reach normalized device coordinates.

Good naming habit

If a variable is named viewPosition, that name should tell you which transforms have already happened and which have not.

Good debugging habit

When something disappears, ask which step likely pushed it out of the valid region instead of treating the full chain as one black box.

Clip Space

Clip space is a temporary homogeneous checkpoint, not the screen

This is the term that most often gets blurred together with NDC or “screen space.” Clip space is earlier than both. It is the projection output before the perspective divide.

View space

The point is camera-relative, but projection has not happened yet.

p_view = (x_v, y_v, z_v)

Clip space

Projection has happened. The point is now homogeneous and still carries w.

p_clip = (x_c, y_c, z_c, w_c)

NDC

The same point after dividing the clip coordinate by w.

p_ndc = p_clip / w_c

Viewport

No new 3D transform here. The GPU maps NDC into pixel coordinates on the screen.

pixel = viewport(p_ndc)

Visual explainer

Projection creates clip space. Dividing by w creates NDC. The viewport then maps that normalized box into pixels.

  1. Projection outputs a homogeneous clip coordinate

    You still have four numbers there, and w is one of the important ones.

  2. Visibility is tested against w

    For a point to survive clipping, the clip-space values have to stay inside the range -w to +w.

  3. NDC is the post-divide coordinate

    After dividing by w, the point lands in the fixed [-1, 1] box. That new post-divide value is NDC, not clip space.

  4. The viewport maps NDC to pixels

    Only after the divide does the GPU map the normalized box into actual screen coordinates. The viewport is a screen mapping step, not another 3D transform.

Clip test -w <= x_c, y_c, z_c <= w
NDC step p_ndc = p_clip / w
Viewport step pixel = viewport(p_ndc)

Step by step

Follow a single vertex through the pipeline

  1. Begin in object space

    The vertex lives in the model’s own local coordinates. Nothing about the scene or camera matters yet.

  2. Apply the model matrix

    This moves the model into world space: translation, rotation, and scale usually happen here.

  3. Apply the view matrix

    The whole world is re-expressed from the camera’s frame so “forward,” “left,” and “up” become camera-relative.

  4. Apply projection

    Perspective or orthographic projection shapes the visible volume into clip space. That clip coordinate still carries w. Only after dividing by w do you get normalized device coordinates.

The full chain

Most tutorials write this as P * V * M * p_h. Read it right to left as a sequence of spaces ending in a divide by w.

Clip versus NDC

Clip space is the projection output that still carries w. NDC is the new coordinate you get after dividing by w. The viewport then turns NDC into pixels.

Why names matter

If a variable is called worldPosition, that name should tell you what operations are still valid on it.

Debugging habit

When geometry disappears, ask which stage likely pushed it out of the visible volume.
