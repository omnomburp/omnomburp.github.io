Acceleration, culling, and level of detail

This chapter shifts from image formation to cost control: deciding which parts of the scene deserve work, which can be skipped, and which can be simplified.

Chapter 15

Fast rendering is often about refusing to do unnecessary work

A good frame is not only about correct shading. It is also about scale. Modern scenes contain more objects, more lights, and more effects than a naive pipeline can afford to process blindly. Acceleration structures, frustum culling, and level-of-detail systems are all ways of asking a disciplined question before expensive work happens: does this object still matter enough for the current view?

Culling

Skip objects that are outside the relevant view or otherwise unable to contribute to the final image.

Acceleration

Organize scene data so large groups can be rejected quickly instead of testing every object one by one.

Level of detail

Spend high geometric detail only where the camera is close enough for that detail to matter.

Frustum test dot(plane_normal, center) + radius > 0
LOD selection screen_size = radius / distance

Game intuition

Every shipped game depends on not rendering what cannot be seen

Open-world games with thousands of objects, dense city scenes with millions of triangles, and battle royale maps with 100 players all rely on acceleration and culling to hit their frame budgets. Without these systems, even powerful GPUs would choke on the sheer submission cost of objects that contribute nothing to the final image.

Open worlds

A large terrain with forests, buildings, and NPCs. The camera sees a tiny fraction at any moment. Frustum culling and LOD keep the visible budget manageable.

Interior scenes

Rooms connected by hallways. Occlusion culling skips entire rooms the camera cannot see through doorways, saving both GPU and CPU cost.

Ray tracing acceleration

A ray tracer without a BVH tests every ray against every triangle. With a BVH, most triangles are skipped after a few box tests, making real-time ray tracing possible.

Streaming and LOD

Distant mountains load low-poly meshes. As the player approaches, higher detail streams in. LOD systems manage this transition so pop-in is minimized.

Mental model

Performance systems decide what deserves detail before shading happens

1

Start with a view

The camera defines a volume and a distance range where objects could plausibly matter.

2

Reject what cannot contribute

If a bound never overlaps the view, the renderer should avoid paying full submission or intersection cost for it.

3

Simplify what contributes only weakly

Distant objects still matter, but not enough to justify the same mesh density as the hero object near the camera.

4

Track the cost in scene-space terms

Triangle counts, instance counts, and traversal depth are often better diagnostics than "the frame feels slow."

Deep dive

A BVH organizes objects into a tree of bounding boxes for fast rejection

A Bounding Volume Hierarchy (BVH) is a tree structure where each node contains a bounding box that encloses all objects in its subtree. The root box contains the entire scene. Each internal node splits its objects into two groups, each with its own tighter bounding box. To test a ray or frustum against the scene, start at the root: if the query misses the root box, the entire scene is skipped. If it hits, descend to the children and repeat. Most branches are rejected early, so a scene with millions of triangles can be queried with only tens of box tests. The key insight is that missing a parent box lets you skip all the children at once.

A ray traverses the BVH. Green boxes are hit and explored further. Red boxes are missed and their entire subtree is skipped. The triangle hit count stays low because most of the scene is rejected by box tests.

  1. Build phase: split objects into groups

    Choose a split axis and position that produces two groups with tight bounding boxes. Common strategies: surface area heuristic (SAH), median split, or spatial median.

  2. Each node stores only a bounding box and child pointers

    Leaf nodes point to actual geometry. Internal nodes just define the box and link to children. The tree is typically 10-20 levels deep for millions of triangles.

  3. Traversal tests the ray against boxes, not triangles

    A ray-box test is cheap (a few comparisons). Only when a leaf is reached does the ray test against actual triangles. Most triangles are never tested.

BVH vs grid vs octree

BVHs adapt well to non-uniform object distributions. Uniform grids waste memory in sparse areas. Octrees are a compromise but BVHs are the standard for ray tracing.

Cost model

Without a BVH, testing N triangles costs O(N). With a balanced BVH, average cost is O(log N). This is the difference between real-time and minutes per frame.

Scene management

Acceleration structures and LOD systems are both forms of triage

  1. Bounding volumes are cheap summaries

    A box or sphere does not describe the full object, but it is good enough for many early reject tests.

  2. Frustum culling limits submission cost

    If an object is outside the view, the best fragment shader is the one you never had to run.

  3. LOD limits overdrawn detail

    When an object occupies only a few pixels, a high-resolution mesh is often wasted effort.

  4. Acceleration scales both rasterization and ray tracing

    Raster pipelines cull and batch aggressively. Ray tracers rely on traversal structures so every ray does not test every triangle.

Common failure mode

The scene renders correctly, but off-screen objects still get processed because nothing ever tells the pipeline to stop early.

LOD warning

Detail transitions can pop visibly if the swap thresholds are too sudden or if shading does not match across levels.

Broad-phase intuition

Acceleration structures are useful because most potential tests are irrelevant. The structure exists to skip them.

Good debugging move

Render bounding volumes and LOD tiers directly. If you cannot see which system picked an object, you cannot trust its cost model.

Concept isolation

The same object at different detail levels balances quality against cost

LOD systems store multiple versions of the same mesh at decreasing triangle counts. The renderer picks the version whose detail matches the object's screen-space size. A character at 2000 triangles looks the same as one at 20,000 when it occupies 30 pixels on screen, but costs 10x less to render. The challenge is making the transitions invisible: blending between levels, matching silhouettes, and ensuring shading stays consistent.

Three versions of the same object. High detail (left) for close-up views, medium (center) for mid-range, and low (right) for distant rendering. Triangle counts shown below each.

Concept isolation

Frustum culling tests an object's bounding box against six planes

The view frustum is defined by six planes: near, far, left, right, top, and bottom. To test whether an object is visible, check its axis-aligned bounding box (AABB) against each plane. If the box is entirely on the outside of any plane, the object is outside the frustum and can be skipped. The test is a few dot products per plane, making it very cheap compared to actually rendering the object. The demo below shows a single object with its AABB and the six frustum plane tests.

The bounding box is tested against each frustum plane. Green planes pass (box is on the inside). Red planes fail (box is outside). One red plane is enough to cull the entire object.
