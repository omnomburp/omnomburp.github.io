Chapter 12

Transparent images care about order, edges, and how layers mix

Opaque rendering is mostly a winner-takes-the-pixel story. Transparency breaks that simplicity. Now the renderer has to care about draw order, partial edge coverage, how alpha is interpreted, and whether the colors being blended make sense for the pipeline around them.

Transparency

Alpha says how much a layer should contribute, but it does not remove the need to reason about order.

Antialiasing

Coverage near an edge should change smoothly if you want silhouettes and overlaps to feel stable.

Compositing

The final pixel is a layered result, not just a single surface color.

Straight alpha over opaque bg out_rgb = src_rgb * a + dst_rgb * (1 - a)
Premultiplied alpha premul_rgb = rgb * a
Sorting rule draw far layers before near layers

Game intuition

Where transparency shows up in real-time scenes

Transparency is not just stained glass. Nearly every shipped game and real-time application has to deal with layered, partially see-through surfaces. The same compositing decisions drive particle systems, UI overlays, foliage cards, and hair rendering.

Particle effects

Smoke, fire, and spell effects layer many semi-transparent quads. Wrong draw order makes the stack look inverted or causes bright halos around dark particles.

Foliage and hair

Tree leaves and hair strands use alpha-tested or alpha-blended cutouts. Sorting thousands of tiny blended cards per frame is a real production cost.

UI and HUD layers

Health bars, minimaps, and dialog boxes blend over the 3D scene. Getting the alpha math wrong produces dark outlines or washed-out text on varied backgrounds.

Water and glass

Refractive and transmissive surfaces need to composite the background they reveal. A glass panel with wrong ordering looks opaque from certain angles.

Mental model

One pixel can be the result of several partial contributions

1

Coverage is geometric

A triangle edge may cover only part of a pixel. Antialiasing tries to preserve that partial coverage instead of snapping to all or nothing.

2

Opacity is material

A tinted window can fully cover a pixel geometrically while still letting background light show through.

3

Order changes the answer

Blend two translucent layers in the wrong order and the same scene data produces a different final color.

4

Correct compositing is part of image quality

An otherwise sharp renderer can still look cheap if transparency halos, jagged silhouettes, or sorting errors remain unresolved.

Concept isolation

Coverage and opacity are related ideas that solve different problems

Coverage asks: how much of this pixel does the triangle actually occupy? Opacity asks: even where the triangle fully covers the pixel, how much light does it let through? A triangle edge at the boundary of a pixel has partial coverage. A glass pane in the center of a pixel has full coverage but partial opacity. Antialiasing addresses the first; alpha blending addresses the second.

Left: a triangle edge partially covers a pixel grid (geometric coverage). Right: a fully covering surface with adjustable material transparency (opacity). Both affect the final pixel color, but they come from different sources.

Order rules

Transparency stops being a pure depth-buffer problem

  1. Opaque geometry can usually trust the depth buffer

    The nearest fragment wins, and later fragments behind it can be rejected cheaply.

  2. Translucent geometry often needs back-to-front blending

    The background color must survive long enough for each layer in front of it to mix correctly.

  3. Aliasing is especially obvious on translucent edges

    Harsh binary edges create shimmering silhouettes and make overlapping layers look unstable.

  4. Premultiplication and color space still matter

    If alpha math is correct but the colors being mixed are wrong for the working space, the final image still looks off. The simple source-over formula above assumes straight alpha over an opaque background; premultiplied-alpha pipelines store alpha in the source color already.

Common failure mode

A glass object is drawn before the wall behind it, so the wall never contributes correctly to the final pixel.

Edge reminder

Coverage says how much of the pixel belongs to the primitive. Opacity says how much light that primitive should contribute.

Where MSAA helps

It improves geometry edge coverage. It does not magically solve every transparency ordering problem by itself.

Good debugging move

Freeze the camera, disable one translucent layer at a time, and inspect whether the blend order matches the scene depth you expect.

Deep dive

Premultiplied alpha stores the multiplication up front so compositing gets simpler

In straight alpha, a pixel stores its full RGB color and a separate alpha channel. The blend formula multiplies rgb * a every time it composites. In premultiplied alpha, the source color is already premul_rgb = rgb * a before it reaches the compositor. This changes the blend formula from out = src * a + dst * (1 - a) to the simpler out = src + dst * (1 - a). Premultiplied alpha composes more cleanly when layering many passes, avoids dark fringe artifacts on partially transparent edges, and is the internal format most GPU compositors expect.

Left: straight alpha compositing. Right: premultiplied alpha compositing. Watch for dark fringe differences at the semi-transparent edges.

  1. Straight alpha: color and coverage are separate

    RGB stores the full color. Alpha is multiplied in at blend time: out = src_rgb * src_a + dst_rgb * (1 - src_a).

  2. Premultiplied alpha: color already includes coverage

    RGB is pre-scaled by alpha. Blending simplifies to out = src_premul + dst * (1 - src_a). No extra multiply needed.

  3. Premultiplied avoids edge fringing

    When filtering or mipmapping semi-transparent textures, straight alpha can produce dark borders because RGB values near transparent pixels were never meant to be seen at full strength.

GPU default

Most real-time compositors (including WebGL and GPU blend units) expect premultiplied source colors for correct multi-layer results.

Conversion

To convert: premul = vec4(rgb * a, a). To reverse: straight = vec4(premul.rgb / a, a) (guard against dividing by zero).

Deep dive

MSAA estimates edge coverage by testing multiple sample points per pixel

Multi-Sample Anti-Aliasing (MSAA) is a hardware-accelerated technique for reducing jagged edges on geometry silhouettes. Instead of shading every sub-pixel, MSAA evaluates the triangle coverage test at multiple sample positions within each pixel (typically 2, 4, or 8 samples). The fragment shader runs once per pixel, but the coverage mask records which samples the triangle actually covers. The final pixel color is weighted by how many samples passed. This smooths geometry edges without multiplying the shading cost by the sample count.

A zoomed pixel grid with sample points. The triangle edge crosses some pixels partially. MSAA counts how many samples each pixel covers to estimate edge softness.

  1. Multiple sample positions per pixel

    Each pixel contains 2, 4, or 8 sample points at fixed offsets. The rasterizer tests each point against the triangle.

  2. Coverage mask determines blend weight

    If 3 out of 4 samples are inside the triangle, the pixel gets 75% of the fragment color and 25% of the background.

  3. Shading runs once, not per sample

    The fragment shader executes once per pixel. Only the coverage test is per-sample, keeping the cost manageable.

MSAA limitation

MSAA only smooths geometry edges (triangle boundaries). It does not help with aliasing inside the shader, such as high-frequency texture patterns or specular highlights.

Transparency interaction

Alpha-to-coverage can use MSAA sample masks to approximate order-independent transparency for alpha-tested surfaces like foliage.
